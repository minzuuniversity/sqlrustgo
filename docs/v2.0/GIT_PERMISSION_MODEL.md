# Git & GitHub 角色与权限体系设计

## 一、总体治理结构（Hierarchy Model）

### 🎯 设计目标

- **权限分层**：开发权、合并权、发布权分离
- **发布权独立**：最高权限账号仅用于发布
- **开发权与合并权分离**：开发者不能直接合并到主干
- **AI 只能处于最低权限层**：AI 永远不能接触最高权限账号

```

  │
  ├── release/*

  └── dev
       │
       └── feature/*
```

### 分支职责

| 分支 | 职责 | 权限 |
|------|------|------|
| **main** | 生产级代码 | 仅 yinglichina 合并 |
| **release/*** | 发布准备分支 | 仅 yinglichina 操作 |
| **dev** | 集成分支 | 仅 maintainer 合并 |
| **feature/*** | 功能开发分支 | 开发账号工作区 |

## 四、GitHub 分支保护设计

### 🔴 main 分支保护（最高级）

- ✅ Require pull request
- ✅ Require at least 1 review
- ✅ Require status checks to pass
- ✅ Require conversation resolution
- ✅ Require signed commits
- ✅ Restrict who can push → 仅 yinglichina
- ✅ 禁止 force push
- ✅ 禁止删除

### 🟡 dev 分支保护

- ✅ Require pull request
- ✅ Restrict who can push → 仅 maintainer
- ✅ 禁止 force push

### 🟢 feature/* 分支

- ✅ 允许开发账号 push
- ✅ 禁止直接 merge main

## 五、合并队列（Merge Queue）设计

### 目标

避免以下问题：

- PR 竞争合并
- CI 通过后被后续 PR 破坏
- main 不稳定

### 规则

#### 1️⃣ feature → dev

1. maintainer 触发合并
2. 启动 CI
3. CI 通过后自动合并

#### 2️⃣ dev → main

1. yinglichina 批准
2. 启动完整测试矩阵
3. 自动进入合并队列
4. 依次串行进入 main

### 合并队列策略

- 只允许单一 PR 进入 main
- 进入队列后冻结 rebase
- 自动生成 changelog
- 自动打 tag（但不发布）
- 发布需手动确认

## 六、AI 不可绕过的权限模型

> 这是 2.0 的核心。

### 原则一：AI 永远处于 L1 或 L2

- AI 可控制 openheart / heartopen
- 可选控制 maintainer（如果你允许）
- **永远不能控制 yinglichina**

### 原则二：最高权限账号物理隔离

**yinglichina 账号要求：**

- 独立 SSH key
- 独立目录
- 不存储 token
- 不配置 gh auth
- 启用 2FA
- 不用于本地开发

### 原则三：目录级身份自动匹配

AI 启动时必须执行：

```
1. 读取当前路径
2. 自动匹配 git user.name
3. 禁止使用 --global
4. 若路径包含 yinglichina → 终止
```

### 原则四：提交前强制验证 Hook

在所有 L1 / L2 目录配置 `.git/hooks/pre-commit`：

```bash
#!/bin/bash
# 校验路径
# 校验 git user.name
# 校验当前分支合法性
# 不匹配立即阻断
```

### 原则五：主干不可本地 push

**main 分支规则：**

- 本地禁止 push
- 只能通过 PR
- PR 必须经过 CI + Review

## 十、核心定义（白皮书级别）

> SQLRustGo 2.0 采用分层治理模型，将开发权、分支治理权、发布权彻底解耦。
> 通过 GitHub 分支保护、合并队列与身份隔离，实现 AI 不可越级的安全架构。
> 主干分支始终处于最高权限账号控制之下，任何自动化系统均不得直接接触发布权。
